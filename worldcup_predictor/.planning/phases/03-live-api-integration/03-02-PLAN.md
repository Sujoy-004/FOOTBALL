    **Addition 2: validate_api_key() function** — Add as a module-level function (before main()):

    - Signature: `def validate_api_key() -> str`
    - Read `os.environ.get("FOOTBALL_API_KEY")`.
    - If None or empty: print `"Error: FOOTBALL_API_KEY environment variable not set."` to stderr, print `"Get a free API key at https://www.football-data.org/"` to stderr, `sys.exit(1)`.
    - Make a test GET request to `"https://api.football-data.org/v4/competitions/WC"` with header `{"X-Auth-Token": api_key}` and timeout `10`.
    - If status == 403: print `"Error: Invalid FOOTBALL_API_KEY (HTTP 403)."` to stderr, `sys.exit(1)`.
    - If status != 200: print `"Warning: API availability check returned {status_code}, continuing..."` to stderr (non-fatal — the API might be down but key is valid).
    - Return the `api_key` string.
    - Per D-16 security: Never log the key value itself. The error messages above only say "Invalid FOOTBALL_API_KEY" or reference the env var name.

    **Addition 3: Integrate into main() flow** — Insert after `played = state.load_played()` and before `probs = run_simulation(...)`:

    1. Call `validate_api_key()` to get `api_key`. If it exits, the error messages are printed and the process terminates (D-16 fail fast).

    2. Call `aliases = state.load_aliases()` and store.

    3. Call `raw = fetch_raw_matches(api_key)`.

    4. If `raw` has elements:
       Call `new_matches = process_matches(raw, teams, bracket, aliases, set(played.keys()))`.
       For each `match_record` in `new_matches`:
       - Update Elo: `ratings_update = elo.update_ratings(match_record["team_a"], match_record["team_b"], match_record["winner"], current_elos)` where `current_elos = {name: data["elo"] for name, data in teams.items()}`.
       - Apply rating changes: `teams[name]["elo"] = new_rating` for each team in ratings_update.
       - Save to played: `played[match_record["match_id"]] = match_record`.
       Print info: `f"  New result: {match_record['team_a']} {match_record['home_score']}-{match_record['away_score']} {match_record['team_b']}"`.
    5. If `raw` is empty (API failure or no new matches), skip processing with an info message.

    6. After fetcher processing, persist any changes:
       `state.save_teams(teams)` and `state.save_played(played)` — but ONLY if new_matches was non-empty (avoid unnecessary writes).

    **Error handling:**
    - Wrap the fetcher block in `try/except Exception as e`: log the exception via `print(f"Warning: Fetcher error: {e}", file=sys.stderr)` and continue. The fetcher should never crash the main loop (D-13 fallback behavior).
    - The existing outer `try/except ValueError/FileNotFoundError/JSONDecodeError` block already handles startup errors.

    **Print statements to add (for diagnostic output):**
    - After alias loading: `print(f"Loaded team aliases: {len(aliases)} entries")`
    - After fetch call: `print(f"Fetched {len(raw)} matches from API")`
    - If new matches detected: `print(f"New matches processed: {len(new_matches)}")`
    - If raw empty: `print("No new matches from API (cached data will be used)")`

    **Edge cases:**
    - `FOOTBALL_API_KEY` not set → validate_api_key exits 1 with clear instructions
    - API returns [] → print "No new matches from API" and proceed to simulation with existing data
    - API call raises exception → caught by try/except, print warning, proceed to simulation
    - Some matches match bracket, others don't → process_matches handles per-match filtering

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Environment → main.py | FOOTBALL_API_KEY env var loaded into process memory |
| main.py → stderr/stdout | Error messages may accidentally leak API key |
| main.py → state.py save | New match records persisted to played.json |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Information Disclosure | validate_api_key error messages | mitigate | Error messages reference the env var name (`FOOTBALL_API_KEY`) not its value. "Invalid" / "not set" — never include the key string in output. Logged at warning level only. |
| T-03-05 | Denial of Service | API key validation call | accept | Single GET to /v4/competitions/WC on startup with 10s timeout. Low impact — one extra request per process lifetime. |
| T-03-06 | Elevation of Privilege | save_teams / save_played writes | mitigate | Written to DATA_DIR only (controlled constant). Atomic write pattern prevents partial writes. No user-controlled path injection. |
| T-03-07 | Spoofing | No API key at all | mitigate | D-16: validate_api_key exits 1 if FOOTBALL_API_KEY is missing. Prevents silent operation without auth. |
| T-03-SC | Tampering | package installs | mitigate | No new packages in this plan. Only stdlib additions (`logging`, `os`) and existing `requests`. |
</threat_model>
