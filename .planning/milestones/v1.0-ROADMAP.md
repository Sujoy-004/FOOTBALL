# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-06-14
**Phases:** 1-6
**Total Plans:** 10

## Overview

A live, self-updating tournament predictor in your terminal — when a match ends, within seconds the script detects it, updates Elo, re-simulates, and shows how every team's odds changed.

## Phases

### Phase 1: State & Elo Foundation

**Goal**: Tournament bracket and Elo ratings can be loaded, computed, validated, and persisted across restarts via JSON files.
**Depends on**: Nothing (first phase)
**Plans**: 2 plans

Plans:

- [x] 01-01: Data Loading & Bracket Validation (scaffold, state.py load+validate, seed data, main.py entry)
- [x] 01-02: Elo Engine & State Persistence (elo.py, atomic save functions, integration test)

**Details:**
- K_FACTOR=60, DEFAULT_ELO=1500 constants
- `state.py`: load_teams, load_bracket, load_played, save_teams, save_played with atomic write
- `elo.py`: expected_score (standard Elo formula), update_ratings (pure function, no mutation)
- Bracket validation: unique match_ids, valid source_matches, no circular dependencies
- 32 team seed data in teams.json, 23 bracket matches in bracket.json

### Phase 2: Monte Carlo Simulation

**Goal**: System can simulate the remaining knockout tournament 50,000+ times and compute accurate championship probabilities from current Elo ratings.
**Depends on**: Phase 1
**Plans**: 2 plans

Plans:

- [x] 02-01: Core simulation engine (simulation.py + tests + main.py wiring)
- [x] 02-02: Performance benchmark (benchmark script + verification)

**Details:**
- Pure-functional Monte Carlo engine with no I/O side effects
- `run_simulation(teams, bracket, played, iterations=50000, seed=None)` — main entry point returning dict
- Round-specific sim: `_simulate_r16`, `_simulate_knockout_round` (shared for QF/SF/FINAL)
- 50K iterations in 1.268s (PASS, 3.7x headroom over 5s gate)
- Reproducible seeds via `random.seed(seed)` inside the function

### Phase 3: Live API Integration

**Goal**: System fetches live match results from Football-Data.org API with robust error handling — retries on failure, cached data fallback, and never crashes due to network issues.
**Depends on**: Phase 1
**Plans**: 2 plans

Plans:

- [x] 03-01: Core fetcher module + unit tests (fetch_raw_matches, process_matches, 9 test cases)
- [x] 03-02: main.py integration + API key validation

**Details:**
- `fetch_raw_matches(api_key, api_url, timeout)` — HTTP GET with manual 3-retry loop (1s/2s/4s backoff)
- Handles HTTP 429 with Retry-After header (or 60s default)
- `process_matches(raw, teams, bracket, aliases, played_ids)` — resolves API team names via alias lookup
- `_build_alias_lookup(aliases, bracket)` — two-phase build: auto-populates all 32 bracket team names as self-aliases, then overlays team_aliases.json entries
- `validate_api_key()` — checks FOOTBALL_API_KEY env var, test-calls API, exits 1 on missing/403
- 9 tests covering success, empty, retries, 429, timeout, malformed JSON, normalization

### Phase 4: Main Loop & Shutdown

**Goal**: System runs autonomously — polls continuously, detects new matches, triggers re-simulation, and shuts down gracefully on Ctrl+C.
**Depends on**: Phase 3
**Plans**: 1 plan

Plans:

- [x] 04-01: Main loop, signal handlers, graceful shutdown

**Details:**
- `while _running:` loop with signal handler (SIGINT, SIGTERM, SIGBREAK on Windows)
- First poll fires immediately on startup (no initial 60s wait)
- Next-poll calculation prevents drift when fetch/sim takes variable time
- `_next_poll_sleep(interval)` — deadline-based sleep in 0.5s increments, responsive to Ctrl+C within 500ms
- Ctrl+C sets `_running = False`, finishes iteration, saves state, prints final probabilities, exits
- Hourly auto-resim: if `time.time() - last_sim_time > 3600` with no new matches, re-simulates without API call
- Rate limiter prevents hitting Football-Data.org's 10 req/min limit

### Phase 5: Console Output & Formatting

**Goal**: System displays beautiful, colored, delta-tracking championship probabilities in the terminal — readable at a glance with rich formatting and plain-text fallback.
**Depends on**: Phase 2
**Plans**: 2 plans

Plans:

- [x] 05-01: Core probability table with ANSI colors, delta tracking, and risers/fallers
- [x] 05-02: Remaining output blocks (header, match alerts, heartbeat, shutdown, errors)

**Details:**
- Raw ANSI escape codes (no colorama dependency — pure stdlib)
- `_ansi()` factory pattern returns functions: `_bold_cyan`, `_green`, `_red`, etc.
- `_supports_color()` checks `sys.stdout.isatty()` (not cached — handles pipe after TTY)
- Delta column inline in table: ▲ green, ▼ red, = flat, — for new
- Biggest Risers and Biggest Fallers as separate block below table (top-3 each)
- `print_error()` to stderr
- All lifecycle events get timestamp prefix via `_timestamp()` — dim gray when colored
- Windows support: `os.system("")` for Console Host ANSI, `sys.stdout.reconfigure(encoding="utf-8")` for Unicode symbols
- 23 tests across 11 test classes

### Phase 6: CLI Interface & Polish

**Goal**: User controls the tool via command-line flags with full usage documentation — one-off runs, color control, reproducibility, and help on demand.
**Depends on**: Phase 5
**Plans**: 2 plans

Plans:

- [x] 06-01: Argparse + --help + --no-color (argparse framework, NO_COLOR flag, tests)
- [x] 06-02: --once + --seed (single-cycle mode, reproducibility propagation, tests)

**Details:**
- `_parse_args(argv=None)` with argparse — exposes --once, --no-color, --seed N, --help
- `output.NO_COLOR = False` module-level flag; `_supports_color()` checks `sys.stdout.isatty() and not NO_COLOR`
- `--once`: straight-line fetch→simulate→print→exit(0), no signal handlers, no state save
- `--seed N`: threaded through `_run_iteration()` to every `run_simulation()` call
- `--help`: shows all 4 flags with descriptions plus detailed epilog
- 13 CLI tests (8 argparse + 3 NO_COLOR + 2 integration)

## Milestone Summary

**Key Decisions:**

- Python 3.10+ with minimal dependencies (requests only)
- Football-Data.org free API (10 req/min)
- Elo rating with K=60 for championship simulation
- Monte Carlo simulation at 50K iterations
- JSON file persistence with atomic writes
- Console-only output (no web UI for MVP)
- Pure stdlib ANSI (no colorama)
- `--once` mode skips signal handlers and state save
- `--seed` applies to every simulation call, no global `random.seed()`

**Issues Resolved:**

- Bracket DAG validation catches circular dependencies and duplicate match_ids
- Team-alias matching resolves ~91% of API team-name mismatches
- Performance: 50K sim in 1.268s (3.7x headroom)
- Windows ANSI support via `os.system("")` + `sys.stdout.reconfigure(encoding="utf-8")`

**Issues Deferred:**

- End-to-end verification with real Football-Data.org API key
- Group stage simulation (deferred to v2)
- Web dashboard (deferred to v2)

**Technical Debt Incurred:**

- `test_main_loop_runs_iterations` checks for "Fetched" which was renamed to "Polling" in Phase 5
- Team-name alias matching could break on unexpected API response formats
