<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI argv `--seed N` | Untrusted integer input crosses into simulation state via random.seed() |
| CLI argv `--once` | Execution mode switch — no loop branch vs polling loop branch |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | DoS | --seed with large value | accept | `random.seed()` accepts any hashable; Python `int` is unbounded but performance impact of large seed is negligible (hash + setstate) |
| T-06-05 | Tampering | --once skip shutdown save | accept | Intentional design (D-01): --once does not save state because no state changed. User gets fresh snapshot on every --once invocation. |
| T-06-06 | Spoofing | --seed sets global random state inside simulation | mitigate | `random.seed()` is called INSIDE `run_simulation()` with the function's local seed parameter — previous global state restored on each call via fresh seed (per D-04: no global random.seed() at module level) |
| T-06-SC | Tampering | simulation.random.seed | accept | `random` is CPython stdlib — no legitimacy concern ([ASSUMED]) |

## Safety Note: --once Signal Handling

`--once` mode intentionally skips signal handler registration (D-02). If the user presses Ctrl+C during a --once cycle, Python's default SIGINT handling raises `KeyboardInterrupt`, producing a stack trace. This is acceptable for MVP: --once cycles complete in under 2 seconds typically (50K sim in ~1.3s). The stack trace is informative for debugging.
</threat_model>
