  ### A. `_historical_catch_up()` function

  ```python
  def _historical_catch_up(
      data_dir: str,
      elo_ratings: dict[str, float],
      played: dict[str, dict],
      elo_applied: list[str],
  ) -> tuple[dict[str, dict], list[str]]:
      """Fetch all prior finished BSD matches on first run and process chronologically.
      
      Args:
          data_dir: data directory path
          elo_ratings: mutable team ratings
          played: mutable played matches dict
          elo_applied: mutable list of applied match_ids
          
      Returns:
          Updated (played, elo_applied)
      """
  ```
  
  **Logic:**
  1. Check if `played` is non-empty → if so, return immediately (catch-up already done).
  2. Fetch historics from BSD API (`league_id=7`, `date_from` = start of season 2025-06-01).
  3. Filter to finished matches only (`status == "finished"`).
  4. Sort chronologically by `event_date`.
  5. For each match, call `apply_elo_update()`, add to `played` dict, track in `elo_applied`.
  6. After processing all, save state via `save_ucl_played()` and `save_ucl_elo_applied()`.
  7. Print summary: `"Historical catch-up: ingested N prior match(es)"`.
  8. Return updated `(played, elo_applied)`.

  **Error handling (per D-03):** Retry with exponential backoff (up to 3 times: 1s, 2s, 4s). If all retries fail, log error and return `(played, elo_applied)` unchanged (skip catch-up, retry on next poll).

  **Edge cases:**
  - API returns 0 matches (off-season or wrong league_id) → print message, return empty
  - Partial failure: some matches processed before network error → state already partially persisted; next run retries
  - elo_applied already contains some match_ids → skip those (restart guard)
